Research Brief: mini-graph-card, SwitchBot Meter Plus / Indoor‑Outdoor Thermo‑Hygrometer, and HACS Installation#

This brief consolidates factual, sourced information for a technical blog post combining the kalkih/mini-graph-card custom Lovelace card with SwitchBot temperature/humidity sensors in Home Assistant, installed via HACS.

1. mini-graph-card (kalkih/mini-graph-card)#

1.1 What it does#

Mini Graph Card is a minimalistic, customizable graph card for the Home Assistant Lovelace UI authored by kalkih. It works with entities from the sensor and binary_sensor domains and renders both the entity’s current state and a line (or bar) graph representation of its recorded history. The project is MIT‑licensed and at the time of writing the latest stable release is v0.13.0 (released 29 May 2025), with roughly 3.7–3.8k stars on GitHub.

Conceptually, the card retrieves history from Home Assistant’s recorder, groups recorded values into time buckets defined by group_by and points_per_hour, and then collapses each bucket into a single point or bar using an aggregate function (e.g., avg, min, max).

1.2 Key features#

Drawing from the official README, the card offers:

  • Multiple entities on one graph, with per‑entity colors, names, units, and aggregate functions.
  • Line, bar, and “fill” rendering modes, with optional smoothing and configurable line width.
  • Dynamic line coloring via color_thresholds (with optional linear interpolation of stops and smooth/hard transitions).
  • Primary and secondary Y‑axes, including soft/hard upper and lower bounds and a min_bound_range to prevent tiny variations from being exaggerated.
  • Flexible time windows through hours_to_show and points_per_hour (governing graph detail/smoothing).
  • Aggregate functions: avg, median, min, max, first, last, sum, delta (max − min), diff (last − first).
  • State display options including the current state, average, extrema, legend, Y‑axis labels, and per‑entity show/hide of state, line, fill, points, and legend.
  • Show/hide UI elements (name, icon, state, graph, fill, points, legend, average, extrema, labels, labels_secondary, loading_indicator).
  • Tap actions (more-info, navigate, call-service, url, none).
  • Local browser caching of history (cache: true by default) and a configurable update_interval (in seconds) instead of every state change.
  • Logarithmic scale, state_map for non‑numeric/binary sensors, value_factor scaling, and attribute access including dot‑notation traversal of dictionary and list attributes.

1.3 Installation via HACS — exact steps#

The official README states the card is available in HACS (Home Assistant Community Store), noting that “HACS is a third party community store and is not included in Home Assistant out of the box.”

The standard HACS flow for this card is:

  1. Open Home Assistant → HACS in the sidebar.
  2. Open the Frontend section (in HACS 2.x this list is unified under “Custom repositories”/the main HACS view, but the card appears in the same place either way).
  3. Click Explore & Download Repositories (the “+” button at the bottom right).
  4. Search for “mini-graph-card” and select the entry by kalkih.
  5. Click Download, confirm the version in the dialog, and click Download again. HACS places the bundle at config/www/community/mini-graph-card/mini-graph-card-bundle.js (referenced in Home Assistant as /hacsfiles/mini-graph-card/mini-graph-card-bundle.js).
  6. Restart Home Assistant (or reload the frontend resources) so the new card is registered.
  7. Add the resource reference if it is not added automatically (HACS does this for most modern installs, but the README gives explicit instructions for both YAML and UI):
    • YAML: in configuration.yaml, 
      resources:
  - url: /hacsfiles/mini-graph-card/mini-graph-card-bundle.js
    type: module
    • UI: enable advanced mode in your user profile, then go to Settings → Dashboards → Resources (the README phrases this as Configuration → Lovelace Dashboards → Resources Tab), click the orange +, set URL to /hacsfiles/mini-graph-card/mini-graph-card-bundle.js, and select type JavaScript Module.
  8. Add a manual card in your dashboard with type: custom:mini-graph-card.

The README explicitly notes for HACS users: “Use /hacsfiles/mini-graph-card/mini-graph-card-bundle.js and select ‘JavaScript Module’ for HACS install.”

1.4 Key configuration options for this post#

The following are taken directly from the project’s README option tables.

Option Type Default Scope Purpose
show_state boolean — Entity (and show: block) Display the entity’s current state. Set true on a per‑entity basis when graphing multiple sensors so each value is visible in the legend area. In show:, the corresponding state flag accepts true, false, or last (where last shows the value of the last graph point).
color_thresholds list of {value, color} objects — Card Defines threshold stops for dynamic graph coloring. With color_thresholds_transition: smooth (default) values blend; with hard, the color switches abruptly. Intermediate stops can omit value, in which case they are interpolated linearly between the surrounding stops. Useful for things like coloring a humidity graph green between 40–60% and red below or above.
y_axis: secondary string — Entity Routes that entity to the right‑hand secondary Y‑axis instead of the default left axis. Pair with lower_bound_secondary, upper_bound_secondary, min_bound_range_secondary, and show.labels_secondary for proper scaling. The README’s Alternate y‑axis example mixes a “Night” indicator on the secondary axis with three temperature sensors on the primary axis.
decimals integer — Card Forces the exact number of decimals shown for displayed states (does not affect data resolution, only rendering).
hours_to_show integer 24 Card How many hours of history the graph covers. Common values: 24 (1 day), 72 (3 days), 168 (1 week).
aggregate_func string avg Card or per‑entity Method used to collapse each time bucket into a point: avg, median, min, max, first, last, sum, delta, diff. Set per‑entity to plot, e.g., daily min/max/avg of the same temperature sensor on one graph.
points_per_hour number 0.5 Card Number of data points to display per hour — effectively the detail / smoothing of the graph. Higher values (e.g., 60) give near‑realtime detail for short windows; lower values (e.g., 0.25) smooth out long windows like a week of data.

The README’s Show data from the past week example uses hours_to_show: 168 with points_per_hour: 0.25; the Data aggregation functions example combines per‑entity aggregate_func: max/min/avg with group_by: date; and the Dynamic line color example shows color_thresholds for a temperature sensor:

color_thresholds:
  - value: 20
    color: "#f39c12"
  - value: 21
    color: "#d35400"
  - value: 21.5
    color: "#c0392b"

2. SwitchBot Meter Plus and Indoor/Outdoor Thermo‑Hygrometer in Home Assistant#

The smarthomeindex.com URL referenced in the brief was not directly retrievable, so the information below is drawn from SwitchBot’s official product pages, the official Home Assistant SwitchBot documentation, and independent reviews (SmartHomeScene, The Gadgeteer, Matter Alpha, Creating Smart Home, TechHive).

2.1 What sensors do they provide?#

SwitchBot Meter Plus (indoor thermo‑hygrometer):

  • Temperature and relative humidity, sampled by a Swiss‑made (Sensirion) sensor at roughly 4 readings per second.
  • Manufacturer‑quoted accuracy: ±0.2 °C / ±0.4 °F for temperature and ±2% RH for humidity (the broader datasheet range cited by SwitchBot is up to ±1.8% RH and ±0.36 °F).
  • A 3‑inch monochrome LCD with a comfort indicator (dry/comfort/wet) and battery icon.
  • Derived metrics — dewpoint, vapor‑pressure deficit (VPD), and absolute humidity — are computed by the SwitchBot app/cloud and require a SwitchBot Hub for push notifications based on those derived values.

SwitchBot Indoor/Outdoor Thermo‑Hygrometer (Meter Outdoor):

  • Temperature and relative humidity from the same Swiss Sensirion sensor with a 4‑second sampling interval.
  • Specified operating range −20 °C to 60 °C; SwitchBot indicates it can read down to −30 °C / up to 80 °C in practice (per the Creating Smart Home review).
  • IP65 dust/water resistance, AAA × 2 batteries, manufacturer‑quoted up to ~2 years battery life, lanyard included for hanging.
  • No built‑in display — readings are seen via a paired Meter Plus / Hub 2 / Meter Pro display, or via an app/integration. Same dewpoint/VPD/absolute humidity metrics when paired with a SwitchBot Hub.

When integrated into Home Assistant via the SwitchBot Bluetooth integration, the entities exposed for these meter‑class devices are typically: temperature, humidity, battery, and Bluetooth signal strength (RSSI). (The Meter Pro CO2 Monitor adds a CO₂ sensor, and certain users have reported the Meter Plus battery sensor as occasionally unavailable — see the linked Home Assistant Community thread — but temperature and humidity work reliably.)

2.2 Native Home Assistant integration#

The official SwitchBot integration in Home Assistant is a Bluetooth integration. Per the official documentation:

  • “In order to use this integration, it is required to have working Bluetooth set up on the device running Home Assistant. A SwitchBot Hub is not required for this integration.”
  • The Meter, Meter Plus, Indoor/Outdoor Thermo‑Hygrometer, and Meter Pro (CO2 Monitor) are all auto‑discovered by Home Assistant as Bluetooth devices, exposing temperature, humidity (and CO₂ for the Meter Pro CO2), plus battery and signal strength.
  • Bluetooth range can be extended using Home Assistant Bluetooth proxies (e.g., ESPHome on an ESP32, or a Shelly Bluetooth Gateway), which is the recommended approach for outdoor sensors placed beyond the range of the host’s Bluetooth radio.
  • A SwitchBot Hub is only required for cloud features such as remote access, long‑term cloud history, derived‑value alerts, and Matter exposure (via Hub 2 / Hub 3 / Hub Mini Matter).

The Home Assistant Community forum has historically also documented an alternative MQTT path (e.g., sbm2mqtt) that scrapes BLE advertisements from a Pi and publishes them to MQTT, but the project README itself notes the official Bluetooth integration is now the recommended, simpler method:

“UPDATE — The Home Assistant SwitchBot Bluetooth integration now supports Meters. That along with Bluetooth proxies is a much simpler way to bring your Meters into Home Assistant. I now recommend that you use that method instead.” — Home Assistant Community, “SwitchBot Meter Temperature & Humidity … with Home Assistant”.

2.3 Bluetooth, local polling, and cloud#

SwitchBot meters communicate over Bluetooth Low Energy (BLE 4.2 or above) with a manufacturer‑specified open‑air range of roughly 120 m / 394 ft. In Home Assistant, the integration uses passive BLE advertisements by default, with an “active” Bluetooth mode available for devices that store some data (such as battery level for some meters) only in service data accessed via a connection.

Crucially, no cloud connectivity is required: data flows device → Bluetooth → Home Assistant locally, which means readings keep working during internet outages and remain on your network. The SwitchBot app/cloud is optional and only adds remote access, longer history retention, derived‑value push alerts, and Matter bridging via a Hub.

For sensors using SwitchBot’s encrypted BLE protocol (Lock, Lock Pro, certain newer devices) Home Assistant requires a key ID + encryption key, which can either be entered manually or imported by signing into a SwitchBot account through the integration. The basic Meter and Meter Plus do not require this; the Meter Pro CO2 Monitor is one of the encrypted models.

2.4 Why these meters are a good choice for an HA dashboard#

Synthesizing the SwitchBot product pages and the third‑party reviews:

  • Local, hub‑less Home Assistant integration over BLE — no SwitchBot Hub required for temperature/humidity/battery to appear in HA.
  • Industrial‑grade Sensirion sensors with quoted accuracy of ±0.2 °C / ±2 % RH (Meter Plus) and 4 Hz sampling.
  • Bluetooth proxy support enables placement well beyond the host’s BLE range.
  • Long battery life on AAA cells (~1 year for Meter Plus, up to ~2 years for the Indoor/Outdoor model in SwitchBot marketing) and IP65 rating on the outdoor unit, making it suitable for terraces, greenhouses, and even sauna/freezer testing per Creating Smart Home.
  • Affordable — the Indoor/Outdoor Thermo‑Hygrometer retails at around USD $14.99 / EUR €19.99, the Meter Plus at a similar price point.
  • Independent reviewers (SmartHomeScene, The Gadgeteer, Matter Alpha) consistently rate the readings as accurate against calibrated reference sensors (SmartHomeScene reports an average offset of about 0.35 % vs. a calibrated Xiaomi LYWSD02).

Caveats to note in the post: SwitchBot’s marketing tightly couples advanced features (push alerts, dewpoint/VPD push notifications, cloud history > 68 days, Matter) to a SwitchBot Hub, so a hub is needed if you want any of those — it is not needed for raw HA integration. Also, derived metrics like absolute humidity/dewpoint/VPD are not exposed natively as separate Home Assistant entities by the BLE integration; they are computed in SwitchBot’s app/cloud and would have to be calculated in Home Assistant via templates (e.g., a template sensor using temperature and humidity) if needed there.

3. HACS overview — installing a custom card via the Frontend section#

3.1 What HACS is#

The Home Assistant Community Store (HACS) is a third‑party add‑on/integration for Home Assistant that provides a UI for discovering, installing, and updating community custom integrations, frontend cards/themes, and other resources hosted on GitHub. It is not included in Home Assistant out of the box; users install HACS first (via the official documentation at hacs.xyz) and authorize it with a GitHub device login.

3.2 Installing a custom card (general flow)#

Once HACS is installed and configured, the general procedure to install a custom Lovelace card is consistent across community guides (JuanMTech, BforBenny, the HACS docs) and matches the kalkih/mini-graph-card README:

  1. Open HACS from the Home Assistant sidebar.
  2. Open the Frontend section (in HACS 1.x this is a tab labeled “Frontend”; in newer HACS releases all categories are visible from the main HACS screen).
  3. Click Explore & Download Repositories (the orange + button, typically bottom‑right).
  4. Search for the card by name (e.g., mini-graph-card, mushroom, button-card) and select the matching entry.
  5. Read the repository description page that opens, then click Download.
  6. In the version dialog, leave the latest version selected and click Download again. HACS pulls the JS bundle into config/www/community/<repo-name>/, exposed at /hacsfiles/<repo-name>/<bundle>.js.
  7. Restart Home Assistant (or at minimum reload the frontend) so the new resource is loaded by Lovelace.
  8. Verify the resource is registered under Settings → Dashboards → Resources. Modern HACS versions add the resource entry automatically; if it is missing, manually add a JavaScript Module resource pointing at the /hacsfiles/... URL.
  9. Add the card to a dashboard either through the visual card picker (the new card type will appear in the “Custom” section once cached) or by editing YAML and using type: custom:<card-name> (for example, type: custom:mini-graph-card).

If the card does not appear in the picker right away, clear the browser cache and hard‑reload — this is a frequently reported gotcha in the Home Assistant Community (“HACS and mini_graph-card”, “Mini-graph-card Setup” threads).

3.3 Updating and removing#

HACS shows updates for installed cards on its main dashboard; clicking Update downloads the new bundle and you simply refresh the browser (sometimes restart Home Assistant) for it to take effect. Removing a card is similarly done from HACS, after which you should also delete the manual resource entry in Settings → Dashboards → Resources if it was added manually, and remove any custom: cards from your Lovelace YAML to avoid render errors.

Sources of conflict / caveats noted while researching#

  • The kalkih/mini-graph-card README in places still shows pre‑2023 navigation labels (e.g., “Configuration → Lovelace Dashboards → Resources Tab”) which in modern Home Assistant is Settings → Dashboards → Resources. The functionality is identical; only the menu wording has changed.
  • HACS has restructured its UI multiple times. Community guides written before mid‑2024 often refer explicitly to a “Frontend” tab; recent HACS versions consolidate categories on a single page but the install procedure (Explore & Download Repositories → search → Download → restart) is unchanged.
  • Some older Home Assistant Community posts describe MQTT‑based or REST/cloud‑based integrations of SwitchBot meters (sbm2mqtt, the OpenAPI rest_command approach). These predate, and are now superseded by, the official SwitchBot Bluetooth integration in Home Assistant, which the maintainers and reviewers explicitly recommend.
  • A handful of users report intermittent unavailable battery sensors specifically on the SwitchBot Meter Plus with the BLE integration; temperature and humidity still update normally. This is documented in the Home Assistant Community thread “Switchbot Meter Plus battery sensor unavailable” but is not reported for the regular Meter or the Indoor/Outdoor model.
  • SwitchBot’s own marketing pages quote both “±0.2 °C / ±2 % RH” (Meter Plus product page) and “±0.36 °F / ±1.8 % RH” (Indoor/Outdoor product page) for what is essentially the same sensor family; these are the manufacturer’s own figures, not independently verified, so the post should attribute them as such.
  • The exact device referenced by https://smarthomeindex.com/?deviceId=cac98b86-c500-4aea-bbe6-81ffad793cb6 could not be confirmed by direct retrieval during this research; the Meter Plus / Indoor‑Outdoor Thermo‑Hygrometer details above are sourced from SwitchBot’s official product pages and reputable independent reviews instead.
